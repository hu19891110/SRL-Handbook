Sphere Research Labs - official handbook

This document is the guide for users of our online judge based services including:
  • Sphere Engine™
  • Sphere Judge™
  • Sphere Contest™

First of all it is not comprehensive and exhaustive manual for vast area of technical issues of mentioned services. We focus on the essence of functionality namely the idea of processing source codes submitted into the system. As the owner of one of our products you will be creator of new programming problems. Let as introduce the convention of naming - the person who creates the problem is the problem setter or the author and the person who use the service and solves the problem is just the user. All presented information are independent of the type of our service thus we just call it service. Also note that the services are rather oriented to deal with console applications with standard input/output communication. For more definitions please look at the dictionary appending.

1. Introduction

To sketch the idea as simple as possible let us consider the classical method of correctness verification for the source code. The mentor needs to try the code manually i.e. he needs to try it for some simple input data. In that situation even simple programs require at least small human readable interface. Let us consider the following toy example along with the C language solution:

Integer power
Write the program which takes two numbers a and b and returns the value of ab.
int atob(int a, int b)
{
  int i, result = 1;
  for(i=0; i < b; i++)
  {
    result *= a;
  }
  return result;
}

int main()
{
  int a,b;
  printf("Give a: ");
  scanf("%d", &a);
  printf("Give b: ");
  scanf("%d", &b);
  printf("a to the power of b = %d", atob(a,b));
  return 0;
}

After compiling and runing that piece of code the mentor can use the program. Nevertheless, it is easily seen that manual verification requires continuous preparation - for each input the mentor must know the correct output. That's the rule for all problems he manages. Another problem is that he probably wants to be reliable thus he needs to handle with all boundary, clever and problematic input instances. Some interesting situations can be pointed even for that simple problem above. For example, is a to the power of zero gives 1?

If it somehow is not good motivation for automatic judge we can ask another question: how to deal with time and memory complexity issues? Academic experience of members of our team yelds that the automatic judge solutions save plenty of time and greatly improve the quality of verification.

Moving on to the online judging we will present you how we need to change approach. Now whole the process of the compiling, runing and testing going to be conducted by the machine thus we need to clearly specify how the program should communicate - that part we call input/output specification. Suppose we want to prepare the Integer power problem to automatic judging and suppose the presented source code fulfills desired input/output specification. It means we need, along with the problem definition, give the information that program expects two integer numbers on the input and needs to print the information "Print a: Print b: a to the power of b = ab". Clearly the output specification is highly unnatural.

As you can see we don't need to have any human readable interface at all. In fact, it is rather obstruction which can even leads to reject correct solutions due to some complicated or nonintuitive input or output requirements. How simple can the previous Integer power problem could be if we drop all the input / output formating? We don't want to have any "Give me the value" statements to obtain the input and "Result is" statements in the output thus it would be enough to stay with minimal source code:

int atob(int a, int b)
{
  int i, result = 1;
  for(i=0; i < b; i++)
  {
    result *= a;
  }
  return result;
}

int main()
{
  int a,b;
  scanf("%d %d", &a, &b);
  printf("%d", atob(a,b));
  return 0;
}

At the end we want to clearify that the input / output considerations have nothing in common with the length of user's source code. It's only related to the number of input / output operations. If you are experienced programmer it's probably obvious but we put such emphasis on that because of our experience with online judge beginners users. It's the most dangerous when you deal with programmers who have some basic knowledge already and along with that probably some habits. We highly recommend you to also put emphasis on input / output specific when you will present your instance of the service to your own users. More formal and deep description of input and output is one of the most important part of the next section.

Finally, let us take a look at the simplified submission flow diagram:

In this diagram we can see how user's source code go through the system. Firstly it is compiled by a proper compiler (according to the selected programming language) what produce executable file1. In the next step created executable is fed with base input data thus we obtain some user's output file. The final step is the procedure of comparing user's output with the base output, we call it judging. The judge returns the status of the submission. Note that the diagram is a simplified sketch of the idea and we will extend both the picture and the explanation.

2. Problems and submissions

In this section we are going to give you insight into submission flow and online judge problem construction. From the previous section we know basics of the concept of the online judging. To better understand and make it possible to create own problems we will describe components of the problem:

  • Description
    • The task
    • Input / output specification
    • Examples
  • Test case(s) (possibly many)
    • Input file, Output file
    • Time limit
    • Judge
  • Master judge

Before we begin the analysys of the individual components let us take a loot at the complete diagram of submission flow to embed the listed components in overall view.

The diagram is the extension of the one from the previous section. The main difference is the appearance of the master judge component. Since we have introduced possiblity of many test cases (pairs input / output files) thus we need the summary component to combine the results from test case judges - we call that summary component master judge. The other new elements of the diagram was mentally present before but we purposely omited them to keep the view as simple as possible. Now we can explain that every submission need to precise programming language to choose proper compiler or interpreter. The problem description is the crucial part for users to make it possible to solve the problem.

Now let us recall the Integer Power problem since we are going to use it as a demonstration example. The probem is to compute the value of ab for a given integers a and b.

Description

This is the only important feature from the user's point of view. Users must be able to understand the problem to solve it successfully.

The task

The formal text describing the specificity of the problem usually in mathematical manner. To achieve good quality of the description it is a good habit to graphically illustrate the problem. It is also desirable to support a dry theory with simple examples.

According to the Integer Power problem we could write the taks as follows:

Write the program which takes two numbers a and b and returns the value of ab. For example for a = 3 and b = 4 the result is 34 = 81

Input / output specification

The previous part was formal from the mathematical side but not syntactically. Automatic judging is possible when we can make an assumption on the problems's input and output behaviour. The rest is a black box for us right now. We specify precisely what will be the input file structure to make possible for users to implement proper input gathering. The specification of input file should be an information that user can rely on.

On the other hand we expect that user produce output file which structure is also predictable. Formal specification of the output file is an information that we are going to rely on to verify the correctness of the submission.

According to the Integer Power problem let us recall possible solution of the problem. We didn't specify any formal input or output syntax in the previous section. First attempt was the solution with the human readable interface and after that we modified the code to achieve a raw version. We can use that source code as a base for following formal specification:

Input: In the only line of the input there will be two integer numbers 1 ≤ a ≤ 8 and 0 ≤ b ≤ 10 separated by a single space character.

Output: Program should write a single number which is a value of ab.

In practice the examples here are dedicated to ilustrate that input and output files structure. In the best case scenario they cover every distinct configuration of parameters (up to numbers, letters etc.) which is important for more complicated problems.

Test case(s)

Jast as the description was for user's only so the test cases are for the machine checker. This is the essence of the idea. The vast majority of real life usages implements the following schema: there is a pair of input and correct output and the script or the program which can compare that correct output with the user's output to decide whether user's answer is good or not.

Input and output file

The pair of corresponding files. Input file contains the instance of the problem and it must be consistent with the input specification. The output of the correct program should be contained in the output file. It is not necessery to write the solution to the program to create the output file - it can be obtained in any manner.

According to the Integer Power problem we present how we could prepare following test cases:

Test case 1:

  Output file:

Test case 2:
  Input file:
  7 0
  Output file:
  1

Remark. It is recomended to construct the problems that are able to repeat the desired procedure as many times as we want to make it possible to wide test the input instances domain. We will see an example of such popular contruction in the next section.

Time limit

We have already pointed that one of the feature of online judging is the possiblity of estimation the time complexity. To achieve that the author of the problem has to adjust the timeout for program execution. Consider the case when the author knows two different algorithms for a problem, say A and B. Let us assume that the algorithm A is noticeable faster than the algorithm B. It is still not very easy and obvious how to preper the whole test case to distinguish between these two algorithms. However, assuming that we have input data which is processed in the time tA for the algorithm A which is much faster than execution time tB for the algorithm B we can simply set the time limit somewhere between those values.

With the timeout tA ≤ t0 ≤ tB we can assume that A-like algorithms will pass the test case and B-like algorithms will fail it with because of time limit exceeded.

Remark. Note that presented approach highly depends on the machine thus you need to adjust your time limit to the computing cluster rather then your local machine

Our toy example problem is much too simple in assumptions to allow us to present example of time limits that distinguish different algorithms thus we put default time limit of 1s. In the next section we present more complex example where we introduce time limit which can help to estimate the algorithm quality.

Judge

The judge is the program which process user's program output after execution. Its task is to establish if the submission passed the test case and potentially also returns the score.

Usually the judge implementation is reduced to compare the base output with the user's output (possibly with some whitespaces difference ignore). We support problem setters with default judges:

  • Strict - it requires outputs to be identical
  • Ignoring differences in whitespaces - similar to the previous one but it ignores all extra whitespaces
  • Ignoring floating point errors up to a specific position - it allows the floating point numbers to be inaccurate i.e. we can accept the errors up to for example 0.001
Remark. The Ignoring differences in whitespaces judge is one of the most popular default choice. It is more liberal for output formating errors which in fact doesn't affect on the solution semantic correctness. Similarly Ignoring floating point errors up to a specific position judge is popular choice for problems where result numbers are not integers.

We have mentioned that the judge can also returns the score. More information will be presented in the section advanced test case judges, for now you can assume that the score is the test case execution time.

It is possible to create own test case judges. The author can implement any kind of verification having full access to the input file, base input file, user's output file and even user's source code. For more information visit the section advanced test case judges.

For the Integer Power problem we decide to use default Ignoring differences in whitespaces judge for each test case thus we allow the user to generate extra whitespaces before and after the resulting number ab. For example when user prints " 81 " as a result for "3 4" instance it is still correct answer.

Master judge

We have discussed the single test cases for the problem and establish that each of them return some information, say status and the score. The master judge is the component which combines all incoming results obtained from test cases to produce the final result which is the status and the score. You can look again at the submission flow diagram for better understand.

There are predefined master judges proper for most situations:

  • Generic masterjudge - it gathers information from test case judges and requires all of them to be successful to establish final result as the success. When any test case ends with error the final answer is inherited from the first failed test case. For example when the problem have five test cases and the second and the fourth failed the final result is inherited from the second test case. Generic masterjudge combine the execution times of all testcases and yields the sum as the final score.
  • Score is % of correctly solved sets - the more liberal masterjudge which allows to accept incomplete solution with the score which is the percentage of correctly solved test cases. For example when the problem have five test cases and again the second and the fourth failed but the rest was passed then the final score is equal to 60%. The advantage is that the user gets more information about the correctness of its solution.

When you need more complex master judging process or you want to slightly modify the existing ones you can create and upload it. You have access to the source code of default master judges and they can be used as a base for your modifications. Further information about designing master judges you can find in the section Advanced master judges

The last missing part for the example we successively complete is the choice of master judge. We created two test cases and there is no need to implement the specific own master judge thus we select default one. When we need to distinguish the solutions as better or worst (but both correct) we should rather choose Score is % of correctly solved sets but in our situation each test case is a verification of correctness thus we select Generic masterjudge to force the user's solution to pass all test cases.

Submission status

Ultimately, for a full understanding of the diagram let as briefly comment possible statuses which can be assigned to the submission.
  • Accepted (AC) - the submission is a correct solution to the problem.
  • Wrong answer (WA) - the submission is incorrect solution. It depends on master judge implementation but usually if submission failed in at least one test case the answer is wrong.
  • Time limit exceeded (TLE) - the submission execution took to long. Again it depends on master judge but again usually to obtain TLE status from master judge is enough to exceed time in at least one test case.
  • Runtime error (RE) - The error occurred during program execution.
  • Compilation error (CE) - The error occurred during compilation or syntax validation in interpreter.
  • Internal error (IE) - The error occurred on the serivice side. On of the possible reasons can be poorly designed test case judge or master judge.

3. Example

In this section we present more complicated example with full details to give you better overall look at abilities of our services. It is still an elementary example but it tell you much more about the specific of online judging.

The base of the problem is to count the sum of numbers from 1 to given n i.e. 1 + 2 + 3 + ... + n. We extend the problem to handle with multiple input numbers. We call it the Initital sum problem. Look at the following problem description:

For a positive integer n calculate the value of the sum of all positive integers that are not greater than n i.e. 1 + 2 + 3 + ... + n. For example when n = 5 then the correct answer is 15.

Input: In the first line there will be the number 1 ≤ t ≤ 10000000 which is the number of instances for your problem. In each of the next t lines there will be one number n for which you should calculate the described initial sum.

Output: For each n print the calculated initial sum. Separate answers with new line character.

Example 1:
  Input:
  4
  1
  2
  3
  4
  Output:
  1
  3
  6
  10

Example 2:
  Input:
  2
  10
  11
  Output:
  55
  66

We have extened base problem of initial sum calculation to multiple calculation possibility which allows us to distinguis between faster and slower solutions (if we prepare rich test cases). Take into account that the possible input data can be too big for standard int type thus we will use long long type. Before we set test cases let us present two distinct solution to the problem:

#include <stdio.h>

long long initsum(long long n)
{
  int i;
  long long sum = 0;
  for (i=1; i <= n; i++)
  {
    sum += i;
  }
  return sum;
}

int main()
{
  int t;
  long long n;
  scanf("%d", &t);
  while (t > 0)
  {
    scanf("%lld", &n);
    printf("%lld\n", initsum(n));
    t--;
  }
  return 0;
}

The first solution refers to the definition of the problem i.e. the function initsum iterates from 1 to n to calculate desired value. The calculation requires n operations of addition to obtain the result. It is basic school knowledge that there exist the compact formula for that problem and we use it in the second implementation:

#include <stdio.h>

long long initsum(long long n)
{
  return n*(n+1)/2;
}

int main()
{
  int t;
  long long n;
  scanf("%d", &t);
  while (t > 0)
  {
    scanf("%lld", &n);
    printf("%lld\n", initsum(n));
    t--;
  }
  return 0;
}

Both programs are correct answer to the problem but if we want to distinguish the algorithms we can design test cases that only the second solution can pass. As we mentioned before it highly depends on the computational power of the machine. We presend test cases that are valid for our current machines (Intel Pentium G860 3GHz). Our suggestion is to design one test case which is easy to pass for both algorithms to give information that the solution is correct and the second test case that is possible to pass only for the second algorithm. That will give an information that the correct answer can be too slow. The user submitting the solution similar to the first one he gets information about testcases and will be able to see that his program pass first test and exceeded time in the second.

We cannot put all input and output data here because of its size thus we write it in shortened manner:

Test case 1:
  Input file:
  1000
  1
  2
  ...
  1000
  1000000
  Output file:
  1
  3
  ...
  500500
  500000500000

Test case 2:
  Input file:
  1000000
  1
  2
  ...
  1000000
  Output file:
  1
  3
  ...
  500000500000

Computational power of current machines is enough to finish first test case instantly. Both presented algorithms with time below 0.01s. However it is good test case for corectness verification. First 1000 positive integers give us the assurance that solution is mathematically correct. We have also added single test with big number i.e. n = 1000000 to make sure that user's solution bases on long long type. On the other hand the second test case is rich enough to make the first algorithm to exceed even 5s time limit. The second algorithm works fast enough to pass that test case in time below 0.1s. We have huge gap between 0.1s and 5s thus we can easily choose safe value as our time limit, for example again 1s.

We still haven't chose judges for test cases and master judge for the problem. We don't have floating point numbers in our output file specification thus we rather decide to choose Ignoring differences in whitespaces judge for both test cases. It leaves users with possiblity of small formating errors without risk of unwanted rejections of theirs solutions. For example it is possible to separate output answers with spaces instead of new line characters.

We assume that we want to accept every correct solution but distinguish the better once and give them better score. The Score is % of correctly solved sets master judge is perfect for that purpose. Submitting the first solution achieves the result of 50% while the second solution passes both test cases and its result is 100%.

Remark. Presented scoring method assumed both tests as equally worth 50% each. To achieve different distribution of scores you need to modify the master judge and pick the scoring of test cases arbitrary. We present the example in the section advanced master judges.

Appendix 1: Dictionary

  • Sphere Engine™ ­ TODOPlatforma do weryfikacji praktycznych umiejętności programistycznych. Posiada wbudowany zestaw kompilatorów oraz bazę zadań testowych. System pozwala na automatyczną ocenę nadsyłanych rozwiązań, które dostarczane są w postaci kodów źródłowych.
  • Sphere Judge™ ­ TODOPlatforma do weryfikacji praktycznych umiejętności programistycznych. Posiada wbudowany zestaw kompilatorów oraz bazę zadań testowych. System pozwala na automatyczną ocenę nadsyłanych rozwiązań, które dostarczane są w postaci kodów źródłowych.
  • Sphere Contest™ ­ TODOPlatforma do weryfikacji praktycznych umiejętności programistycznych. Posiada wbudowany zestaw kompilatorów oraz bazę zadań testowych. System pozwala na automatyczną ocenę nadsyłanych rozwiązań, które dostarczane są w postaci kodów źródłowych.
  • Problem ­ The challange task for service's users which is the basic element of the service and it contains:
    • Description - text explanation of the problem nature along with input/output specification (see more problem specification)
    • Test cases - the sets of input data along with methods of judging (see more test case)
    • Master judge - the method of final scoring the execution of the submitted solution
  • Test case ­ The set of data (usually pair of input file and correct output file) used to verification of the user's submission. It serves as a certificate of the correctness and also contains:
    • time limit - the maximum time of the user's solution execution with the data from input file
    • tast case judge - the program analyses the output generated by user's solution (usually compares user's output with correct output) (see more test case judge, advance test case judge)
  • Submission ­ The source code which is potentially the correct solution to the problem.
  • Submission status - The short information about the user's submission execution. It can be one of the following: accepted, wrong answer, time limit exceeded, run time error, compile error (see more submission statuses)
  • Contest ­ It is the separated part of the platform's set of problems. It contains subscribed users and individual ranking.
  • User ­ The programmer who has active account on the service.
  • Problem setter ­ The user of the service with the permissions to create new problems and manage problems which he has created.
  • Manager ­ The user of the service with the permissions to manage whole service's content.
  • Administrator ­ The user of the service with full set of permissions.
